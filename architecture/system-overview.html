System Architecture

This document provides a high-level overview of the TAPWORK system architecture, including its core components and their interactions.

Optional Biometric Service

The biometric module is included as an optional service to support future enhancements without affecting current functionality. The system operates fully without it.

Architecture Overviewâ€‹

The TAPWORK system follows a three-tier architecture designed for scalability, maintainability, and future extensibility. The architecture separates concerns into distinct layers, each with specific responsibilities and clear interfaces between them.

Key Architectural Principlesâ€‹

1. Separation of Concerns

  • Each layer has a well-defined responsibility
  • Changes in one layer minimally impact others
  • Clear API contracts between layers

2. Scalability

  • Stateless API design allows horizontal scaling
  • Database connection pooling optimizes resource usage
  • Container orchestration (Kubernetes) enables dynamic scaling

3. Extensibility

  • Optional services (like biometrics) can be added without system redesign
  • Plugin architecture allows new features to be integrated seamlessly
  • API-first design enables multiple client implementations

4. Security by Design

  • Authentication handled at API gateway level
  • Role-based access control (RBAC) for all endpoints
  • Encrypted data in transit and at rest

System Layersâ€‹

1. Client Layerâ€‹

The client layer represents the users who interact with the TAPWORK system through different interfaces:

Employee/User Interface

  • Primary users who record their attendance
  • Access the system via mobile devices or desktop browsers
  • Interact primarily with the Scanner Application
  • Typical actions: clock in/out, view personal attendance history
  • No special permissions required - read-only access to own data

Admin/HR Portal

  • Management interface for HR personnel and administrators
  • Access to system-wide data and configurations
  • Typical actions: manage employees, generate reports, configure system settings, monitor attendance
  • Requires elevated permissions with role-based access control
  • Can manage user accounts, attendance policies, and system parameters

2. Frontend Layerâ€‹

Scanner Application

The Scanner Application is a progressive web app (PWA) that serves as the primary employee interface:

  • Technology Stack: Built with modern JavaScript frameworks (React/Vue.js)
  • Camera Integration: Uses WebRTC MediaDevices API to access device camera
  • QR Code Processing: Real-time QR code detection and validation
  • Offline Capability: Service Workers enable basic functionality without internet
  • Responsive Design: Works seamlessly on mobile phones, tablets, and desktop computers
  • User Experience: Simple, intuitive interface requiring minimal training

Key Features:

  • Real-time camera preview for QR scanning
  • Instant feedback on successful/failed scans
  • Error handling and user guidance
  • Local caching for improved performance
  • Automatic retry logic for failed submissions

3. Backend Services Layerâ€‹

The backend layer runs on containerized infrastructure (Docker/Kubernetes) and consists of several microservices:

API Service (FastAPI)â€‹

The core of the system, responsible for all business logic:

  • Request Handling: Processes all HTTP requests from clients
  • Authentication: JWT-based token validation for secure access
  • Authorization: Role-based access control (RBAC) implementation
  • Data Validation: Input sanitization and validation using Pydantic models
  • Business Logic: Attendance rules, overtime calculations, late/early detection
  • Integration Hub: Coordinates between database, SMTP, and optional services

Key Endpoints:

  • POST /attendance - Record new attendance entry
  • GET /attendance/{id} - Retrieve attendance records
  • GET /reports - Generate attendance reports
  • POST /auth/login - User authentication
  • GET /employees - Employee management (admin only)

PostgreSQL Databaseâ€‹

Relational database providing persistent storage:

  • Data Models: Employees, Attendance Records, Shifts, Departments, Roles
  • Relationships: Foreign keys maintain data integrity
  • Indexing: Optimized indexes for fast querying
  • Transactions: ACID compliance ensures data consistency
  • Backups: Automated daily backups with point-in-time recovery
  • Performance: Connection pooling and query optimization

Key Tables:

  • employees - Employee information and credentials
  • attendance_records - All check-in/check-out records
  • shifts - Work shift definitions
  • departments - Organizational structure
  • audit_logs - System activity tracking

SMTP Serviceâ€‹

Email notification system for automated communications:

  • Development Environment: Uses Mailhog for email testing without sending real emails
  • Production Environment: Configurable SMTP server (SendGrid, Amazon SES, etc.)
  • Notification Types:
    • Attendance confirmations
    • Late arrival alerts
    • Missing clock-out reminders
    • Weekly/monthly reports
    • System alerts for administrators
  • Template System: HTML email templates with dynamic content
  • Queue Management: Asynchronous email sending to prevent blocking

Biometric Service (Optional)â€‹

Future-ready module for enhanced identity verification:

  • Purpose: Adds an additional layer of security beyond QR codes
  • Technologies: Face recognition, fingerprint scanning
  • Integration: Loosely coupled via API calls
  • Activation: Can be enabled/disabled per location or employee group
  • Privacy: Compliant with biometric data regulations (GDPR, BIPA)
  • Fallback: System continues working if biometric validation is unavailable

Implementation Status:

  • Currently in planning phase
  • Architecture supports seamless integration
  • Will not disrupt existing functionality when added

Component Interactionsâ€‹

Standard Attendance Flowâ€‹

  1. Employee Authentication

    • Employee opens Scanner Application
    • May require initial login (cached for subsequent uses)
    • Scanner App receives authentication token

  2. QR Code Scanning

    • Camera activates and displays live preview
    • Employee presents their personal QR code
    • Scanner detects and decodes QR data
    • Client-side validation ensures QR format is correct

  3. Attendance Submission

    • Scanner App sends POST request to /attendance endpoint
    • Request includes: QR data, timestamp, device info, location (if available)
    • API validates authentication token
    • API verifies QR code authenticity

  4. Business Logic Processing

    • API checks if employee exists and is active
    • Determines if action is clock-in or clock-out
    • Validates against shift schedule
    • Detects late arrival, early departure, or overtime
    • Applies attendance policies (grace periods, rounding rules)

  5. Data Persistence

    • Validated attendance record saved to PostgreSQL
    • Transaction ensures atomic operation
    • Database triggers may update related tables

  6. Confirmation & Notification

    • Success response sent to Scanner App
    • User sees confirmation message
    • Optional email confirmation queued in SMTP service
    • Admin dashboard updates in real-time (if monitoring)

Admin Management Flowâ€‹

  1. Admin Authentication

    • Admin logs into HR Portal
    • Higher-level permissions granted based on role

  2. System Management

    • View real-time attendance dashboard
    • Generate custom reports
    • Manage employee accounts
    • Configure system settings
    • Review audit logs

  3. Data Retrieval

    • API queries PostgreSQL with appropriate filters
    • Results formatted and returned to portal
    • Large datasets paginated for performance

Optional Biometric Flowâ€‹

When enabled, adds an additional verification step:

  1. Scanner App captures biometric data (face image or fingerprint)
  2. Data sent to Biometric Service via secure channel
  3. Service performs identity verification
  4. Result returned to API Service
  5. API proceeds with attendance recording only if verified
  6. Fallback to QR-only mode if biometric service unavailable

Client Layerâ€‹

The client layer represents the end-users interacting with the system:

  • Employee/User: End-users who clock in/out using the scanner interface
  • Admin/HR Portal: Administrative dashboard for system management, reporting, and configuration

Frontend Layerâ€‹

  • Scanner Application: Browser-based progressive web app (PWA) that captures QR codes using the device camera and JavaScript. Provides a responsive interface for both mobile and desktop devices.

Backend Servicesâ€‹

The backend runs on Docker/Kubernetes and includes:

  • API Service (FastAPI): Core REST API handling all business logic, authentication, and data validation
  • PostgreSQL Database: Relational database for persistent storage of attendance records, user data, and system configuration
  • SMTP Service: Email notification system (development environment uses Mailhog for testing)
  • Biometric Service (Optional): Future-ready module for face recognition or fingerprint validation

Data Flowâ€‹

The typical attendance recording flow follows these steps:

  1. Access: Employee opens the scanner application via web browser
  2. Scan: User scans their personal QR code using the device camera
  3. Validation (Optional): Biometric validation can be performed if enabled
  4. Processing: API receives the request via POST /attendance endpoint
  5. Storage: Validated attendance record is stored in PostgreSQL
  6. Notification: Email confirmation sent via SMTP service (if configured)

Design Principlesâ€‹

The TAPWORK architecture follows these key principles:

  • Modularity: Each component can be developed, tested, and scaled independently
  • Extensibility: New features (like biometrics) can be added without breaking existing functionality
  • Containerization: Docker/Kubernetes deployment ensures consistency across environments
  • API-First: Clear separation between frontend and backend enables multiple client implementations
  • Security: Authentication and authorization handled at the API layer
  • Scalability: Horizontal scaling supported for all stateless services

Technology Stackâ€‹

Frontendâ€‹

  • Framework: React/Vue.js
  • PWA: Service Workers for offline capability
  • Camera Access: WebRTC MediaDevices API
  • QR Processing: jsQR or similar library

Security Considerationsâ€‹

  • Authentication: JWT-based authentication for all API requests
  • Authorization: Role-based access control (RBAC)
  • Data Encryption: TLS/SSL for data in transit, encryption at rest for sensitive data
  • API Rate Limiting: Protection against abuse and DoS attacks
  • Input Validation: Strict validation of all user inputs
  • Audit Logging: Comprehensive logging of all system actions
